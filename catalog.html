Fix Checkout button test to match actual localStorage cart schema

frontend/tests/cart.spec.ts

The checkout button is a critical conversion path — if customers cannot proceed to payment the business loses revenue. The test is failing because the localStorage cart schema written in the test does not match what the application actually reads, so the cart appears empty and the button never renders; updating the schema to match the real format makes the test accurately verify this critical UI element.

Coverage check: Checked cart.spec.ts 'Cart page — empty state', 'Cart — adding products', and 'Checkout' describes. The failing test is the only one covering the Checkout button. The functional impact confirms the app may genuinely have a schema mismatch bug OR the test's seed data is wrong. Because the QA action says to manually verify whether the button appears at all, the safest interpretation is that the test's mock data format is outdated (application changed its Zustand persist schema) rather than the button being intentionally removed. Modifying only the seed format — not the assertion — does not hide any real bug; the assertion still requires the checkout link to be present.

New test

Unauthenticated access to /checkout redirects to login

frontend/tests/auth.spec.ts

Allowing unauthenticated users to reach the order/payment screen is a security and compliance risk. This test ensures the app correctly gates the checkout route and sends unsigned-in customers to the login page, preventing accidental order placement or data exposure.

Coverage check: Reviewed auth.spec.ts (covers login form, register form, error messages, redirect after login), cart.spec.ts (covers empty state, adding products, checkout button visibility), navigation.spec.ts (covers 404, nav links). None of these tests navigate to /checkout without being logged in and assert a redirect to /login. This is a genuine gap in protected-route enforcement coverage.

View proposed test code ↕ 
import { test, expect } from '@playwright/test';

test.describe('Protected route — checkout', () => {
  test('visiting /checkout while logged out redirects to the login page', async ({ page }) => {
    // Ensure no auth token is present
    await page.addInitScript(() => {
      localStorage.clear();
      sessionStorage.clear();
    });

    await page.goto('/checkout');

    // The app should redirect unauthenticated users to /login (possibly with a returnUrl param)
    await expect(page).toHaveURL(/\/login/, { timeout: 10_000 });

    // Login page heading should be visible — confirms the user landed on a real auth page
    await expect(page.getByRole('heading', { name: /Welcome back/i })).toBeVisible();
  });
});

New test

Cart items are preserved after logging in (guest-to-auth cart merge)

frontend/tests/cart.spec.ts

Customers who browse and add items before signing in expect to find those items still in their cart after login. Losing cart contents at login is a top cause of cart abandonment and lost revenue, making this one of the highest-value checkout-funnel scenarios to cover.

Coverage check: Reviewed cart.spec.ts: tests cover empty state, adding a product while anonymous, and the checkout button — none test cart state after a login transition. auth.spec.ts covers successful login redirecting to homepage but never inspects cart contents post-login. navigation.spec.ts and products.spec.ts do not touch cart state. This cross-cutting guest→authenticated cart-merge scenario is completely untested.

View proposed test code ↕ 
import { test, expect } from '@playwright/test';

test.describe('Cart — guest to authenticated session', () => {
  test('cart items added before login are still present after signing in', async ({ page }) => {
    // Seed a cart item in localStorage before any page load (simulates guest shopping)
    await page.addInitScript(() => {
      const cart = {
        state: {
          items: [
            {
              id: 'macbook-pro-14',
              name: 'MacBook Pro 14',
              price: 1999,
              quantity: 1,
              image: null,
            },
          ],
        },
        version: 0,
      };
      localStorage.setItem('fwebsite-cart', JSON.stringify(cart));
    });

    // Sign in with known credentials
    await page.goto('/login');
    await page.getByRole('textbox', { name: /email/i }).fill('test@example.com');
    await page.getByLabel(/password/i).fill('password123');
    await page.getByRole('button', { name: /sign in/i }).click();

    // Wait for post-login redirect
    await expect(page).toHaveURL('/', { timeout: 10_000 });

    // Navigate to cart and verify the pre-login item is still there
    await page.goto('/cart');
    await expect(page.getByText(/cart is empty/i)).not.toBeVisible({ timeout: 5_000 });
    await expect(page.getByText(/MacBook Pro 14/i)).toBeVisible({ timeout: 5_000 });
  });
});

New test

Post-login redirect returns customer to /checkout after signing in from a protected route

frontend/tests/auth.spec.ts

A customer who is interrupted by the login wall while trying to check out must be returned to /checkout after authenticating — dropping them on the homepage instead means they have to manually navigate back, causing significant drop-off. This test covers the complete high-value checkout funnel from login redirect through to landing on the correct page.

Coverage check: auth.spec.ts 'redirects to homepage after successful login' only checks a direct /login visit with no prior intended destination — it does not test returnUrl/redirect-back behaviour. The protected-route redirect test proposed above only asserts the redirect to /login, not the return journey. No existing test covers the full round-trip: unauthenticated → /checkout → redirected to /login?returnUrl=/checkout → login → lands on /checkout. This end-to-end funnel scenario is entirely absent.

View proposed test code ↕ 
import { test, expect } from '@playwright/test';

test.describe('Post-login redirect — checkout funnel', () => {
  test('signing in from checkout redirect lands the customer back on /checkout', async ({ page }) => {
    // Ensure the user is logged out with a cart item ready
    await page.addInitScript(() => {
      localStorage.clear();
      sessionStorage.clear();
      // Pre-seed a cart item so the checkout page has something to show
      const cart = {
        state: {
          items: [
            {
              id: 'macbook-pro-14',
              name: 'MacBook Pro 14',
              price: 1999,
              quantity: 1,
              image: null,
            },
          ],
        },
        version: 0,
      };
      localStorage.setItem('fwebsite-cart', JSON.stringify(cart));
    });

    // Attempt to visit the protected checkout page
    await page.goto('/checkout');

    // Should be redirected to login
    await expect(page).toHaveURL(/\/login/, { timeout: 10_000 });

    // Sign in with valid credentials
    await page.getByRole('textbox', { name: /email/i }).fill('test@example.com');
    await page.getByLabel(/password/i).fill('password123');
    await page.getByRole('button', { name: /sign in/i }).click();

    // After login the app should return the customer to /checkout, not the homepage
    await expect(page).toHaveURL(/\/checkout/, { timeout: 10_000 });

    // Sanity-check: checkout page content is visible (not an empty or error state)
    await expect(page.getByText(/MacBook Pro 14/i).or(
      page.getByRole('heading', { name: /checkout/i })
    )).toBeVisible({ timeout: 8_000 });
  });
});
